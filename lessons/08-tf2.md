# Lesson 08 — TF2 (coordinate frames)

**The idea:** a robot has many coordinate frames (the map, the robot body, each sensor). **TF2** tracks how they relate **over time**, so any node can ask *"where is frame X relative to frame Y at time T?"* and get an answer — even across a chain of frames.

The standard chain (parent → child):
```
map → odom → base_link → laser / camera / ...
```

## 🔧 For a Go / Java engineer
| TF2 | You know it as |
|---|---|
| the frame tree | a **scene graph** (nested coordinate systems) |
| `/tf` topic + broadcasters | publishers streaming edges of that tree |
| the TF **buffer** | a **time-stamped cache** you interpolate/query |
| `lookup_transform(target, source, time)` | a function that **composes the chain** for you (like a currency converter: "amount in A → value in B") |
| timestamps on every transform | it's a **time-series** — you can ask "where was it 0.5 s ago" |

**Key point:** frames are **timestamped** because sensor data arrives with latency — you must transform a laser reading using the robot pose *at the time the scan was taken*, not now. That time dimension is what makes TF2 more than a static matrix.

## The demo (this package)
- `broadcaster` → publishes `map → base_link` as the robot drives in a 2 m circle (10 Hz).
- `listener` → queries the buffer for `map → base_link` and logs the position each second.

## ▶ Run on Ubuntu
```bash
cd <your ROS2_LEARNING> && git pull
source /opt/ros/jazzy/setup.bash
cd ros2_ws && colcon build && source install/setup.bash

ros2 run tf2_demo broadcaster      # terminal A
ros2 run tf2_demo listener         # terminal B  -> logs the circling position
```
**Inspect from outside (the TF superpowers):**
```bash
ros2 topic echo /tf                                  # raw transforms
ros2 run tf2_ros tf2_echo map base_link              # live transform between any 2 frames
ros2 run tf2_tools view_frames                       # generates frames.pdf — a picture of the tree
```

## 🌍 Real world
Every robot publishes a TF tree. `robot_state_publisher` publishes the fixed sensor↔body transforms from the URDF; odometry publishes `odom→base_link`; localization (AMCL) publishes `map→odom`. **Nav2 will not work without a correct TF tree** — bad TF is the #1 beginner navigation bug. When a laser says "obstacle 2 m ahead," TF is what turns that into "obstacle at map (5.3, 1.1)."

## Takeaways
- TF2 = timestamped scene graph of frames; query with `lookup_transform`.
- Broadcasters publish edges on `/tf`; listeners query the buffer.
- `view_frames` draws the tree; `tf2_echo` shows any pair live.
- Correct TF is a hard prerequisite for Nav2.
